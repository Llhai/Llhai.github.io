HashMap面试100问

Java

发布于：Apr 19, 2022

1. HashMap1.8 做了哪些优化

  1. 引入红黑树来代替长度大于等于 8 的链表，解决了长链表带来的查找效率低的问题
  2. 链表的插入方式从头插法改成了尾插法，解决并发扩容带来的链表成环问题
  3. 扩容的适合 1.7 需要对原数组进行重新 hash 定位在新树组的位置，1.8 采用更简单的判断逻辑，扩容后的新位置只有两种情况，位置不变或者索引+旧容量大小
  4. 1.7 每次插入之前都要判断是否需要扩容，1.8 则是先进行插入，插入完成再判断是否需要扩容

2. HashMap 在线程方面会出现什么问题

  1. jdk1.7 中，在多线程环境下扩容时可能会形成环形链表，导致程序死循环
  2. jdk1.8 中，多线程环境下，会发生数据覆盖的问题

3. HashMap 的底层数组长度为什么总是 2 的幂

  1. 当 length 为 2 的幂时，可以使用 h & (length-1) 来代替 h % length，进行高效的位置计算
  2. 基于 h & (length-1)进行 hash 计算，可以使数据在数组中的分布更均匀，减少冲突

4. 为什么 HashMap 的容量默认值为 16，负载因子默认为 0.75，链表树化的长度阈值为 8

  • 负载因子 0.75：对于负载因子来说，负载因子越小，时间效率越高但空间利用率越高，0.75 可以看作是对时间和空间效率的一个平衡选择，官方作者给出的解释是这样的：

As a general rule, the default load factor (.75) offers a good tradeoff between time and space costs. Higher values decrease the space overhead but increase the lookup cost (reflected in most of the operations of the HashMap class, including get and put). The expected number of entries in the map and its load factor should be taken into account when setting its initial capacity, so as to minimize the number of rehash operations. If the initial capacity is greater than the maximum number of entries divided by the load factor, no rehash operations will ever occur.

  • 树化阈值为 8: 理想情况下，节点在 hash 桶中的频率遵循泊松分布，按照泊松分布计可以计算出桶中元素个数和概率的对照表，当时链表元素个数为 8 时，概率以及很小了
  • 默认容量 16：

5. 计算 hash 值的 hash 扰动

  • JDK1.8 中，hash 扰动的实现是，将 hashcode 异或其右移 16 位的值
  • JDK1.7 中，进行了 4 次扰动，这样降低了计算 hash 的性能

扰动的好处是，可以在数组长度比较小时，让 hashcode 的高位也参与运算，使得节点在数组中的分布更均匀

6. HashMap put 方法的处理流程

  1. 首先计算出新元素的 hash 值，找到在数组中的存储位置
  2. 如果数组为空，则调用 resize() 进行初始化
  3. 如果没有发生 hash 冲突，则直接将元素放在对应的数组下标中
  4. 如果发生了 hash 冲突，且 key 已经存在，则覆盖掉
  5. 如果冲突后，节点是红黑树，则将节点添加到树种
  6. 如果冲突后节点是是链表，则判断该链表是否大于 8 ，如果大于 8 并且数组容量小于 64，就进行扩容；如果链表节点大于 8 并且数组的容量大于 64，则将这个结构转换为红黑树；否则，链表插入键值对，若 key 存在，就覆盖掉 value。
  7. 判断数组是否达到阈值，达到则扩容

9. CurrentHashMap 中的 CAS 和 synchronized 是如何使用的

  • CAS ： 当判断数组中当前位置位 null 时，使用 CAS 来把这个新的 Node 写入数组对应位置
  • synchronized ： 当数组中的指定位置不为空时，通过加锁来添加这个节点进入数组(链表<8)或者是红黑树（链表>=8）
